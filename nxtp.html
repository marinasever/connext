Nxtp - это протокол Connext обеспечивающий полностью надежную передачу данных и вызовов между доменами, совместимыми с Ethereum (домены = блокчейны и L2s).

Полная сеть Connext состоит из мощного стека протоколов. Одним из таких протоколов является Connext nxtp, выполняющий функцию базового протокола, обеспечивающего взаимодействие между сетями.

Nxtp — облегченный протокол
для транзакций Xchain/Xrollup
Connext nxtp разработан как упрощенный протокол. Используется для объединения транзакций xchain/xrollup, которые сохраняют свойства безопасности базовой среды выполнения. Таким образом, он не полагается на внешние валидаторы, а использует набор своих валидаторов в той среде, в которой он работает (например, набор валидаторов Ethereum).
1. Nxtp состоит из простого соглашения, в котором используются шаблоны локинга для подготовки и выполнения транзакций.
2. Сеть автономных маршрутизаторов, участвующих в ценовых аукционах и отвечающих за передачу данных об обращениях между блокчейнами.
3. Протокол также состоит из SDK на стороне пользователя, который осуществляет поиск маршрутизаторов и запроса транзакций в сети.

Жизненный цикл транзакций nxtp
Три фазы, через которые проходят сделки
Транзакции, направляемые через nxtp, проходят в общей сложности три этапа. Рассмотрим эти этапы более подробно:
​
Аукцион маршрутов
Процесс инициируется, в момент, когда пользователь осуществляет широковещательную передачу в сеть Connext, сигнализируя о желаемом маршруте. Затем маршрутизаторы отвечают на эту широковещательную рассылку закрытыми заявками. Эти заявки включают в себя обязательство маршрутизаторов выполнить транзакцию в заранее определенное время и в определенном ценовом диапазоне.
​
Подготовка транзакции
Первый шаг (Аукцион маршрутов) завершается, когда маршрутизатор выигрывает аукцион. На следующем этапе происходит подготовка транзакции. Для этого пользователь отправляет транзакцию в «Менеджер транзакций», заключая договор в сети на стороне отправителя, который включает подписание заявки маршрутизатора. В результате транзакции средства пользователя блокируются в сети отправки.
Как только маршрутизатор обнаруживает событие, содержащее его подписанную заявку, он отправляет ту же транзакцию в «Менеджер транзакций» в сети на стороне получателя. В это же время роутер блокирует соответствующее количество ликвидности. Чтобы маршрутизатор завершил транзакцию, сумма, заблокированная в принимающей сети должна быть равна сумме отправки и аукционного сбора.
​
Выполнение транзакции
Пользователь, инициировавший передачу, посылает запрос при обнаружении события «подготовки транзакции» в сеть на стороне получателя. Сообщение передается ретранслятору, который принимает оплату за транзакцию. Затем сообщение отправляется ретранслятором (который может быть маршрутизатором) в «менеджер транзакций». Транзакция завершается в сети на стороне получателя, затем ретранслятор может потребовать средства, заблокированные маршрутизатором. Ретранслятор используется для того, чтобы пользователи могли отправлять транзакции, содержащие произвольные данные о заявках, в принимающей сети и без необходимости не платить за лишние операции. Исходная сумма будет разблокирована после того, как маршрутизатор успешно отправит то же подписанное сообщение для завершения транзакции на стороне отправителя.
Визуальный обзор жизненного цикла транзакции
Вот обзор, который кратко описывает, как выглядит жизненный цикл транзакции:
Что произойдет, если транзакция не будет выполнена?
Каждая транзакция имеет фиксированный тайм-аут. Как только он будет достигнут, транзакция отменяется. Сторона, которая инициировала подготовку транзакции в сети, может отозвать ее.
До истечения срока действия транзакции она может быть отменена в одностороннем порядке лицом, которому причитаются средства, маршрутизатором в сети отправки или пользователем в сети получения.
Ни одному из участников не требуется хранилище для завершения транзакции. Они могут самостоятельно получать всю необходимую информацию для подготовки (prepare), выполнения (fulfill) или отмены (cancel) транзакций.
